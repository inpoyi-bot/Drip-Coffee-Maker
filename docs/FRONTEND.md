## 1. 色板(6 值,命名 + 语义)

> 骨架五值近乎无彩、微冷;第六值是唯一 signature 跳色。**克制的意义就是让收敛色跳出来**——像 Blue Bottle 全系克制只为那一点蓝。

| token | hex(建议,待审) | 角色 | 用在哪 |
|---|---|---|---|
| `--ink` | `#1A1D21` | 主文本 / 最深 | 正文、标题;微冷近黑,非纯黑(纯黑太硬) |
| `--slate` | `#5B6570` | 次级文本 / 图注 / 数据标签 | confidence 标签、时间轴刻度、辅助信息 |
| `--mist` | `#DDE2E6` | 分隔线 / 边框 / 禁用态 | 细线、卡片边、hairline rule |
| `--fog` | `#F2F4F6` | 次级背景 / 卡片底 | IP 屏卡片、录入区块底 |
| `--paper` | `#FBFCFD` | 主背景 | 微冷白,非暖奶油(刻意避开 #F4F1EA 那类 AI 默认) |
| `--converge` | `#0F9E8E`(青绿) | **★ signature 收敛色** | **仅**用于收敛/满意时刻:满意终止、轨迹收敛点、graduation 达成 |

> **为什么收敛色选青绿(teal)不选蓝**:
> - Blue Bottle 用蓝,直接跟随会显得像模仿其品牌,而非你的选择(skill:别把自由花在别人的默认上)。
> - 青绿在语义上更贴「到位/平衡/健康」(接近医疗仪器的正常读数绿,但更冷静、不刺眼),而蓝偏「信息/链接」。你的收敛色要传达的是**「你到了、这杯对了」**,青绿比蓝更准。
> - 它和微冷骨架同色温,跳出来但不违和。
> ⚠️ 待审:具体色值(青绿深浅)、以及你若偏好蓝可否决此推导。

> **色板红线(接映射表 G2)**:诊断状态的区分**绝不**靠颜色编码承担主要判别——反差对(flavor_mismatch⟂taste_unaddressable 等)必须靠**文案**分开,颜色至多辅助。理由:① 色盲可达性;② 你的 IP 是语义区分不是色彩区分,靠颜色分等于把判别力交给视觉而非诊断。收敛色是唯一被赋予语义的颜色,且只标一个正向状态,不用于区分诊断分支。

## 2. 字体(3 角色)

> Blue Bottle 质感靠细字重 + 宽字距 + 克制,不靠粗黑体。三角色分工,display 用得克制。

| 角色 | 建议字体(待审) | 用在哪 | 性格 |
|---|---|---|---|
| **Display** | 一款有骨感的无衬线(如 Söhne / Neue Haas Grotesk / 备选 Inter Display),字重克制在 Regular–Medium | 屏标题、收敛时刻的宣告 | 精密、冷静,不砸人;**用得少** |
| **Body** | 同族或互补无衬线(如 Inter / Söhne),常规字重 | 对话、诊断文案、正文 | 高可读,中文配思源黑体 / 苹方,保证中英混排不塌 |
| **Mono/Data** | 一款等宽(如 Söhne Mono / JetBrains Mono / IBM Plex Mono) | 数据标签、时间轴刻度、cup 序号、gradient 读数 | **仪器感的关键**——数据用等宽,一眼「这是台会测量的东西」 |

> **★ 等宽字体是仪器气质的承重墙**:轨迹页的杯序、时间、gradient 方向用等宽排,立刻从「聊天 app」变「诊断仪」。这是廉价但高效的 signature 强化。Blue Bottle 不需要数据字体(它不展示数据),但你需要——这是你「仪器 > menu」偏离参照的正当处。
> ⚠️ 中文 demo:display/body 的中文 fallback 要指定(思源黑体系),别让英文字体的中文回退到系统默认宋体破坏冷静感。

## 3. 间距 / 圆角 / 线(基调)

| token | 值(建议,待审) | 说明 |
|---|---|---|
| 间距基数 | 8px 网格(4/8/16/24/32/48) | 秩序感来自一致的节奏,不是随机间距 |
| 圆角 | `--radius-sm: 6px` / `--radius-md: 10px` | 微圆角。**不用零圆角**(太硬/像 broadsheet 默认),**不用大圆角**(太软/像消费 app);中间值贴仪器的「精密但不冷酷」 |
| 边框 | 1px `--mist` hairline | 细线分隔,Blue Bottle 式克制;**绝不**用重阴影堆叠 |
| 阴影 | 极轻或无 | 靠间距和 hairline 分层,不靠 drop-shadow。收敛卡片可例外给一点点提升,强化 signature |

## 4. 分层密度(★ 偏离 Blue Bottle 的正当处)

> Blue Bottle 全屏低密度(它卖咖啡,不展示数据)。你需要两种密度:触点屏亲和,IP 屏承载信息。**同一套 token,两种节奏。**

| 屏类型 | 密度 | 间距节奏 | 目的 |
|---|---|---|---|
| **触点屏**(冷启动问答、每杯反馈录入、对话) | 疏 | 走足留白(24/32/48),大点击区 | 初学者放松、不害怕、不被数据吓到 |
| **IP 屏**(诊断结果、Cup Timeline、我的轨迹页) | 密 | 紧凑但有节奏(8/16),靠对齐+层级组织,不靠留白 | 承载多状态/轨迹/置信,像高级仪表盘不像后台管理系统 |

> 密不等于乱:IP 屏的高密度靠**等宽数据字体 + 8px 网格严格对齐 + hairline 分隔**维持秩序,而非塞满。这是「精密仪器」和「后台管理系统」的分界。

## 5. Signature 收敛时刻(把 boldness 全花在这里)

> skill:Spend your boldness in one place。全 app 克制,唯独「收敛/满意」这一刻给足。

| 触发 | 视觉事件(建议,待审) |
|---|---|
| agent 判定 `satisfied` / 宣布收敛 | 收敛色 `--converge` 首次出现;可配一次克制的过渡动效(reduced-motion 时降级为静态点亮) |
| 我的轨迹页 · 收敛点 | 轨迹终点用 `--converge` 点亮,是全页唯一彩色点;前面震荡/爬升段全用中性色 |
| graduation 达成(萃取毕业) | 同色语义呼应,标记「到位」 |

> **语义留痕**:`--converge` 是**成就标识**不是装饰。用户第一次看到这个颜色 = 他第一次冲对。因此**绝不**把收敛色用在导航/按钮/普通强调等无关处,否则稀释掉「看到它=我到了」的意义。这是比 Blue Bottle 的品牌蓝更进一步的用法:语义化的唯一色。

## 0. 总规则(三档 + 一条兜底)

| 档 | 范围 | 翻译级别 | 理由 |
|---|---|---|---|

## 1. 第一档 · L1 逐值(你的核心 IP,必须上 UI 且互斥)

### 1.1 `terminate_reason`(终止归因 · 最关键)

> 这九个值里 active 七个、悬空两个。**互斥语义在 UI 上绝不能长得像**——这是整个项目 discriminating power 的最终出口。

| terminate_reason | 层级 | L1 中文文案(待审) | 绝不能和谁混 | 状态 |
|---|---|---|---|---|
| `satisfied` | 收敛 | 「这杯你满意了,研磨就调到这儿。想继续优化的话,水温、手法是后面的事。」 | — | ✅ active(E8) |
| `would_overextract` | 收敛(回退) | 「再往细调,反而会变苦。回到上一杯那个设置,那杯最好——研磨就到这里。」 | ⟂ satisfied(一个「够好了」,一个「再动会变苦」) | ✅ active(E9) |
| `plateau_ambiguous` | 到顶(欠定) | 「到这儿再磨细也帮不上了——可能是磨豆机已经磨到最细、再细也没用,也可能是豆子放久了、不那么新鲜了。这两种情况继续磨细都救不回来,所以研磨先停在这。」 | ⟂ 所有「萃取已到位」类 | ✅ active(E5) |
| `axis_unreliable` | 硬件 | 「你这台砍豆机磨出来粗细不均,又酸又苦是它造成的,不是你手法的问题。换成锥刀/平刀磨会稳很多。」 | ⟂ 其它所有(归因在硬件,不在冲煮) | ✅ active(E3) |
| `axis_limit_underextracted` | **萃取还没到位** | 「已经磨到最细可用了,但萃取还没真正到位——偏薄、酸压过甜。研磨这里能调的都调过了,再往细会堵、细粉多、流速不稳。接下来要提升萃取,得靠水温或浓度这些研磨之外的调节,那是下一步。」 | ⟂ taste_unaddressable(**萃取还没到位** vs 萃取已到位) | ✅ active(E12b) |
| `flavor_mismatch` | 萃取已到位 · 风味取向 | 「这杯萃取已经调到位了——剩下的是风味取向。你不太爱这类酸的话,换一支豆会更合你口味,不是冲煮的问题。」 | ⟂ taste_unaddressable(**换一支豆更合口味** vs 同一支豆、下一步调别的) | ✅ active(E11a) |
| `taste_unaddressable` | 萃取已到位 · 研磨之外 | 「萃取到位了,你喜欢这个酸、只是想更厚——这个方向很明确。让它更厚的办法是升温、延长接触、或提高浓度,这些是研磨之外的调节。这一版我们专注把研磨调到最好,那几项留给下一步。」 | ⟂ flavor_mismatch(**同一支豆、下一步调别的** vs 换一支豆) **且** ⟂ axis_limit_underextracted(**萃取已到位** vs 还没到位) | ✅ active(E11b/E11d) |
| `plateau_axis_topped` | — | **不映射** | — | ⚠️ orphan enum:留在总枚举但无 eval 覆盖、无「何种证据才允许用」的定义 → 不分配 UI 语义,意外出现走 §5 fallback |
| `plateau_bean_decay` | — | **不映射** | — | ⚠️ 同上 |

> **设计留痕 · 三组最危险的反差对**(审文案时重点看这三行读起来是否明显不同):
> - `flavor_mismatch`(换一支豆更合口味 · 横向换选择)vs `taste_unaddressable`(同一支豆、下一步调别的 · 纵向进阶)—— E11 命门,方向必须清楚不同
> - `axis_limit_underextracted`(萃取还没到位)vs `taste_unaddressable`(萃取已到位)—— E12/E11 边界,「还没 vs 已经」是根本分界
> - `satisfied`(够好了)vs `would_overextract`(再动会变苦)—— 两种「停」,原因不同
>
> **文案红线 · 清除系统黑话**:用户可见文案**绝不**出现「轴 / 毕业 / 无解 / 口味层 / 萃取层」等系统内部词——按用户认得的命名(磨粗磨细、到位/没到位、换豆、下一步),不按后端概念命名。这些黑话保留在字段名和本表左列(工程侧),**不进右列 L1 文案**。
> **设计留痕 · plateau_ambiguous 的欠定性要故意保留**:文案里「可能磨到最细、也可能豆子不新鲜」两种可能各自**具体说清**,但「到底是哪一种」**不拍**——欠定活在「是哪个」上,不活在「每个是什么」上。两种可能越具体,欠定越可信。忠实 E5,别改成二选一(会 overclaim 一个 agent 做不到的诊断粒度)。

### 1.2 `turn_type`(决策状态)

| turn_type | L1 处理 | 说明 |
|---|---|---|
| `probe` | 上 UI:显示为「agent 正在追问确认」的状态(那句问句本身就是可见动作) | ⟂ terminate。「还在判」不能和「判完了」混 |
| `terminate` | 上 UI:显示为「本轴收敛/停手」状态 | ⟂ probe |
| `adjust` | 上 UI:显示为「这轮调整」 | 与第二档 direction/step 配合渲染 |
| `seed` | 上 UI:显示为「起点配方」 | 冷启动首轮 |

### 1.3 `gradient: 已收敛`(高危 token)

> `已收敛` 是触发口味层的开关。它出现与否必须**准确**,不能因「友好」被模糊。单独锁进 L1:UI 上「萃取已到位/毕业」这个状态只在真正 `已收敛` 时出现。其余 gradient 取值走第二档。

## 2. 第二档 · L2 语义簇(上 UI,可合并,无语义反差)

| 字段 | 取值 | L2 簇化呈现(待审) |
|---|---|---|
| `gradient`(除已收敛外) | 变好+同向 / 变好 / 变坏 / 没变 / n/a | 方向指示:↑ 变好 · ↓ 变坏 · → 没变化 ·(n/a 不显示)。`变好` 可用于 probe 忠实记录本杯相对改善,不代表毕业。不逐字翻译,一个指示符即可 |

| `grind_now` | 自由文本 | 上 UI:「现在磨到:粗砂糖 / …」当前状态提示 |
| `cup_no` / `date` / `axis` | 自动落盘 | 时间轴/序号,天然友好,直接显示 |

## 4. 组件分离 · 用户报告字段(输入)vs 诊断输出字段(翻译)

> 关键区分:下列字段是**用户自己录入的**,不是 agent 的诊断输出。它们属于「反馈录入表单」组件,**不走翻译层**(翻译层只翻 agent 的诊断输出)。列在这里是防止被误并进映射表。

**已决(PM judgment)**:反馈表单走**结构化录入**(下拉/开关/单选),不走自然语言自由填。理由:初学者说不准自己哪里不对(SPEC D1),结构化选项给他脚手架。

**但必须留「不确定」补位——且分两种,不能用一个统一「不确定」糊掉:**

| 补位类型 | 语义 | 下游动作 | 例 |
|---|---|---|---|
| **「没观察」** | 数据缺失(用户压根没看/没留意) | agent 提示「下次留意」,不 probe | 挂粉环:`没注意看` |
| **「形容不出」** | 感官词汇缺失(观察了但分不清) | **agent 该 probe 的触发点** | 酸的收尾:`尝到了但说不清` |

> ⚠️ 这两种补位喂给 agent 的信号不同:「没观察」是请补数据,「形容不出」是启动探针(接 E1b/E11c/E11d 的 admission gate)。**UI 必须让 agent 区分得开**,否则 probe 触发逻辑失明。

| 用户报告字段 | UI 角色 | 结构化选项(含双补位,待审) |
|---|---|---|
| `sensory`(感官词) | 多选 + 补位 | 见下方感官词按钮集 |
| `vs_prev`(和上杯比) | 单选 | 变好 / 变坏 / 没变 / `说不清` |
| `brew_time`(冲煮时间) | 数值录入 | 可填 + `没计时` |
| `bed_shape`(床面形态) | 单选 | 平 / 拱 / 塌坑 / 偏厚下陷 / `没注意看` |
| `wall_ring`(挂粉环) | 单选 | 有 / 无 / `没注意看` |
| `bed_note`(床面备注) | 自由文本 | 选填,补充用 |

**感官词按钮集(已决)**:按「点了之后发生什么」分三类。**按钮要让用户能表达,又不能替 probe 拍板**——所以关键不是给哪些词,是每个词点下去后的行为。

| 按钮 | 类别 | 按钮文案(消歧写清) | 点了之后 | 出处 |
|---|---|---|---|---|
| 酸 | B 触发 | 酸 | → **必然触发**收尾 probe(发空 vs 化甜回甘) | D1/E1b |
| 苦 | A 自足 | 发苦(像烧焦 / 木头味) | 直接落账 | D5/E9 |
| 涩 | A 自足 | 发涩(口腔收紧发干,像浓茶 / 生柿子,**不是苦**) | 直接落账 | D4/E2 |
| 薄 | B 触发 | 薄(寡淡、水感) | → **必然触发**厚薄 / 萃取 probe | D2 |
| 纸板味/闷味 | A 自足 | 有纸板味 / 闷味、香气发闷 | 直接落账 → **前置 E5 陈化信号** | D5·E5 陈化专属词 |
| 尝到了但说不清 | C 补位 | 尝到了但说不清 | → 触发开放 probe | 双补位 |

> **★ 甜不给按钮(已决 · IP 判断)**:感官词集**故意不含「甜」**。理由:E12 的整个设计是「甜够不够格算萃取毕业,不能让用户拍板」——给「甜」按钮就是邀请用户拍板 graduation,和 E12 冲突。甜由 agent 在该问时通过**绝对刻度 probe**主动问(D3),UI 上不放这个诱饵。**绝不**因「完整」把甜加回按钮。
> **★ 纸板味做按钮(已决 · 前置 E5 信号)**:「纸板味/闷味」是 E5 陈化的**专属风味词**(evals.md:纸板感/香气掉,与酸苦的萃取轴不同类)。做成自足按钮 → 把陈化诊断信号前置到输入层,agent 少一轮推断即可走向 plateau_ambiguous 的「豆子不新鲜」分支。
> **A/B 类的判别逻辑**:A 自足=触感/风味明确、点了信号就完整(苦、涩、纸板);B 触发=词本身不可判别、必然要 probe(酸、薄)。这条 A/B 划分**复用**了「主观味觉靠 probe、明确信号可自足」的同一判据。

> **★ 提交形态(已决 · 前端↔agent 接口契约)**:用户点完结构化选项后,前端**把选择组装成自然语言整句**再喂给 agent(如「这杯还是有点酸,比上杯变好,床面平,杯壁没挂粉环」),**不传结构化 payload**。
> **理由(eval-生产同构,关键)**:agent 的 evalset 输入是**自然语言整句**(见 demo-arc 逐字稿 / E1·E11·E12 evalset),agent 是在自然语言输入上被测试和调优的。若生产改传结构化 payload,生产输入分布 ≠ eval 输入分布,eval 全绿不再保证生产行为——引入隐蔽的 eval-生产 gap。**因此接口形态必须跟随 eval 形态。**
> ⚠️ 反向约束留痕:**别为「优雅」把这里改成结构化 payload**。除非 evalset 输入也同步改成结构化,否则此改动会让 eval 失去对生产的保护力。此决定绑定「eval 用自然语言」这一前提;前提变则重判。
> 附带好处:自然语言天然模糊,贴合 SPEC D4 execution gap(相对变化而非绝对参数);结构化 payload 反而易诱导「绝对参数」思维。

## 5. Fallback 规则 · 未映射值兜底

> 因为 `_validate_record_contract` **不校验枚举值域**(只校验字段组合),上游理论上可吐出映射表里没有的值:拼错值、悬空枚举(`plateau_axis_topped`/`plateau_bean_decay`)、或未来新增值。前端必须有兜底,不能崩、不能显示 raw 机器字符串。

**规则:**
1. 前端拿到任一 `terminate_reason` / `gradient` / `turn_type` 值时,先查本映射表。
2. **命中** → 按对应档渲染。
3. **未命中**(未知/拼错/悬空)→ **不显示 raw 字符串**;降级为通用安全文案,例如:「这一轮的判断我先不细分呈现了。」并在开发态 console 记一条告警(便于回溯上游吐了什么)。
4. **绝不**为未映射值即兴编造 UI 语义——尤其悬空枚举:在体验侧假装一个 agent 实际做不到的诊断分支,是一种倒挂 overclaim。

## 6. confidence · 独立视觉(校准维度)

> `confidence`(low/medium/high)承载的是**校准维度**(多有把握),**不是** certainty of diagnosis(能不能判定)。二者是不同的边界(见 evals.md「认知边界 ≠ 校准边界」)。因此 confidence 单独视觉化,**不混进诊断结论那句话**。

| confidence | 视觉(待审) |
|---|---|
| high | ★★★ / 实心可信度指示 |
| medium | ★★☆ |
| low | ★☆☆,附一句「需要更多信息判断」——接 anti-overclaim 设计 |

> 设计留痕:星级只表达 confidence,不表达诊断本身的确定性。附注句定为「需要更多信息判断」——把低置信归因到**信息不足**(客观、指向下一步 probe),而非「我没底气」(主观、显虚)。这更接 anti-overclaim 正意,也和第三档「info_insufficient → 触发 probe」呼应。

## 7. 国际化(i18n)占位 · v1 中文 / v2 预留英文

> **i18n = internationalization(国际化)**,即让软件支持多语言的做法(缩写取首尾字母 i…n,中间 18 个字母)。**v1 只交付中文**;v2 计划加英文。本节规定 v1 该留多薄的占位,不早搭无用设施。

**已决(三留一红线):**

| 决定 | v1 做法 | 理由 |
|---|---|---|
| **文案键值化(留)** | 用户可见文案**不硬编码在组件里**,走键值查找(如 `t('terminate.taste_unaddressable')`)。v1 只填中文列,v2 加英文列 | 低成本占位,省掉 v2 回挖硬编码的大返工 |
| **布局容忍文案变长(留 · 搭车)** | 布局用弹性容器、不写死文案区宽度 | 英文比中文长约 30-40%;此需求与 mobile-first 响应式**高度重叠**,基本免费搭车 |
| **完整 i18n 设施(不留)** | v1 **不做**语言切换器 / locale 检测 / 日期数字本地化 / 复数规则 / RTL | 过早工程化,v1 只有中文用不上;违反「学习>速度、别过度自动化」。v2 需求清楚时再搭 |
| **★ 反差对跨语言守恒(红线)** | 见下 | 防 v2 翻译抹平 IP |

> **★ 红线 · 反差对是跨语言契约,不是中文特供**:i18n 不是「翻译文案」,是「为每个语言各守一遍反差对」。v2 加英文时,`flavor_mismatch`⟂`taste_unaddressable`⟂`axis_limit_underextracted` 等反差对**必须在英文里同样分得开**——**绝不**因翻译图省事把互斥语义都译成模糊的「this cup isn't quite right」。discriminating power 是**跨语言的**,英文译法要复核是否守住 §1.1 的每一组反差对,标准同中文。
> ⚠️ 成本诚实:键值化让文案从「组件里直接写」变成「语言文件里写、组件引用键」,v1 多一层间接。此间接标准且好懂,可接受;但它是 v1 复杂度的真实增量,非零成本。

## 屏 2 · 每杯反馈录入(Cup Feedback Entry)

**这屏的单一职责**:让说不清自己哪里不对的初学者,把这杯的观察结构化录进来。

**关键原子组件**:感官词选择器 · vs_prev 单选 · 冲煮时间 · 床面形态单选 · 挂粉环单选 · **双补位机制**(贯穿各字段)。

| 验收断言 | 必须 / 绝不 |
|---|---|
| 每个客观字段**必须**提供「没观察」补位(如挂粉环 `没注意看`) | 必须(数据缺失 ≠ 强迫瞎选) |
| 感官/主观字段**必须**提供「形容不出」补位(如收尾 `尝到了但说不清`) | 必须(SPEC D1:初学者说不准) |
| 「没观察」和「形容不出」**必须**是**可区分的两类信号**,**绝不**合并成一个统一「不确定」 | 绝不(两者喂不同下游:补数据 vs 触发 probe) |
| 表单**必须**结构化录入(选项/开关),**绝不**要求用户自由描述作为唯一入口 | 必须(D1 脚手架) |

> **这屏是 admission gate 的上游**:「形容不出」这类补位是 agent probe 逻辑(E1b/E11c/E11d)的触发信号。UI 若把它糊进「不确定」,下游 probe 失明。这是本屏最高价值的验收点。

**判别轴 → 输入形态(已决)**:每个维度做成哪种输入,由「用户能否自己分辨」决定——**主观味觉毕业信号靠 probe,客观可视观察上按钮**。

| 判别轴 | 输入形态 | 验收要点 |
|---|---|---|
| **D1 酸的收尾**(发空 vs 化甜回甘) | **(b) 靠 probe** | 给「酸,但说不清收尾」补位触发 probe;**绝不**做成按钮让用户直接选(与 E1b 前提冲突:这正是用户说不清、须 probe 的维度) |
| **D2 体感厚薄** | **(b) 靠 probe** | 薄/厚佐证欠萃 vs 到位,初学者分辨弱,靠 probe 深化 |
| **D3 甜的有无/强度** | **(b) 靠 probe** | 粗粒度可报「尝到一点甜/尝不到」,但**「甜够不够格毕业」必须靠 E12 绝对刻度 probe**,**绝不**用按钮替用户拍板毕业(否则 graduation gate 在输入层被短路,E12 欠萃伪装毕业漏网) |
| **D4 涩的有无** | **(a) 上按钮** | 按钮文案**必须**写清「涩=口腔收紧发干,像浓茶/生柿子,不是苦」 |
| **D5 苦 + 口腔干** | **(a) 上按钮 + 概念折叠** | 按钮文案**必须**把苦、涩分开写清(「发苦,像烧焦/木头味」vs D4 涩);折叠区**必须**补充苦≠涩的区分,不只讲定义 |
| **D6 vs 上杯方向** | **(a) 上按钮** | 变好/变坏/没变 + `说不清`补位;gradient 直接来源 |

> ⚠️ **D4/D5 交界坑(验收重点)**:「口腔发干」同时出现在涩(D4→查通道/豆性)和过萃干(D5→回退),导向不同动作。两个按钮**必须**让苦/涩在输入层就分得开,**绝不**都用光秃秃「发干」——否则 agent 分不清走哪条诊断分支。
>
> **客观信道 O1-O4 默认 (a) 上按钮**:挂粉环、床面形态、流速/冲煮时间、闷蒸排气(E7c 养豆信号)。纯客观观察,无感官分辨门槛,前置省 probe。

## 屏 3 · 诊断结果(Diagnosis Result)

**这屏的单一职责**:把 agent 这一轮的 `record_cup` 诊断输出,按翻译层渲染成用户看得懂的结论 + 下一步。

**关键原子组件**:决策状态标识(probe/adjust/terminate)· L1 终止文案卡片 · L2 方向指示器 · confidence 星级 · rationale 折叠区。

| 验收断言 | 必须 / 绝不 |
|---|---|
| `terminate_reason=taste_unaddressable` 时,**必须**显示「正解在 brew 端 + 本版专注研磨」的**指路**文案,**绝不**出现「换豆/换一支豆」CTA | 绝不(E11b gate 搬到 UI:换掉用户喜欢的好豆是最严重假绿) |
| `terminate_reason=flavor_mismatch` 时,**必须**指向「换一支豆会更对味」,**绝不**说成「本版无解/边界」 | 必须(E11a:可处理,出口在选豆层) |
| `terminate_reason=axis_limit_underextracted` 时,**必须**表述为「萃取还没到位、研磨这里调到头了」,**绝不**表述成「萃取已到位」的口吻、**绝不**出现换豆 CTA | 绝不(E12b:萃取还没到位 ≠ 萃取已到位) |
| `taste_unaddressable` / `axis_limit_underextracted` 文案**必须**是指路语气,**绝不**用「做不到 / 走不动了」等道歉/认输措辞 | 绝不(已决:换框架不换事实) |
| `turn_type=probe` 时,**必须**显示 agent 的追问,**绝不**同时显示「已收敛/停手」终止态 | 绝不(probe⟂terminate) |
| `plateau_ambiguous` 文案**必须**保留「可能极限、也可能豆老」的欠定表述,**绝不**改写成单一确定归因 | 绝不(E5:不 overclaim 做不到的诊断粒度) |
| `rationale` **必须**默认折叠,**绝不**作为主诊断出口 | 绝不(防自由话术抹平反差对) |
| confidence 星级**必须**独立于诊断文案呈现,**绝不**把置信度混进结论那句话 | 绝不(校准维度 ≠ 认知边界) |
| confidence=low 时**必须**附「需要更多信息判断」,**绝不**表述为产品犹豫/没底气 | 必须(anti-overclaim) |

> 这屏验收断言最密——因为它是翻译层的落地点,你整个 IP 的 UI 出口。几乎每条都能直接追回一个 eval case。

## 屏 5 · 「我的」收敛轨迹页(★ demo 高潮,完整版本二)

**这屏的单一职责**:把这包豆的爬山轨迹**画出来**——不是列历史,是可视化 SPEC §3 的收敛循环本身。**这是护城河的单屏物证。**

**关键原子组件**:轨迹可视化(gradient 随杯次变化)· 转折点标注(过冲/回退)· 收敛点高亮 · 终止归因展示 · 跨会话持久化标识。

| 验收断言 | 必须 / 绝不 |
|---|---|
| **必须**按**轨迹**组织(gradient 走向、转折、收敛点可见),**绝不**退化成无结构的历史列表 | 绝不(版本一 vs 版本二的分界:这屏是轨迹不是 log) |
| 收敛/终止那杯**必须**高亮,并显示其 `terminate_reason` 的 L1 文案(G1 同源) | 必须 |
| 出现过冲回退时,**必须**在轨迹上可见「变坏→反向」的转折,**绝不**画成一路平滑上升 | 条件适用(见下) |
| **必须**有跨会话持久化标识(表明「上一杯」来自持久化轨迹,非聊天上下文) | 必须(SPEC §5.3 送命题:这是「为什么是 agent 不是 chatbot」的物证) |
| 轨迹用词**必须**和其它屏同源(G1) | 必须 |

> **为什么这屏值那笔前端预算**:前 4 屏证明「能和 agent 对话」,这一屏证明「它在收敛」。SPEC §5.2 你交付的是**轨迹**不是**点**——这屏就是那句话的字面兑现。面试指着这屏能说清整个护城河。
> **已决:demo 高潮走干净弧线**(demo-arc.md,冷启动→欠萃→一路磨细读梯度→满意停手,无过冲回退)。因此上表「过冲回退」断言在**本次录制不触发**;但**不删**——它保留为条件化断言,若未来改用 dirty arc(demo-arc-dirty-script.md 的杯4 过冲)录制,该断言即生效。留痕:轨迹渲染逻辑本身应能表达转折,只是干净弧线数据里没有转折点。
> ⚠️ scope 诚实:完整可视化较贵。若前端预算告急,降级路径是「轻量轨迹卡片流」(竖排卡片+gradient 箭头+收敛高亮),保住轨迹感的 80%。但当前决定是做完整版。
