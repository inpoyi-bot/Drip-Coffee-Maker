# Probe → Options 清单(草稿 · 待 Chillin 审)

> 用途:每类 agent probe 对应一组**预定义合法选项**(判断二 b),前端渲染成按钮。
> 契约地位:选项走**展示层**,不进 record_cup(判断一 · 加法二),保护 eval。
> **核心原则**:每个选项对应 evals 的判别轴;**"绝不能有的选项"是重点**——那些会破坏判别、让 agent 蒙对的选项必须挡掉。选项措辞和感官词按钮同一性质:承载判别信息,不是随便给。
> 每个 probe 都带一个「说不清」补位(触发继续 probe 或降置信),除非标注不给。

---

## P1 · 冷启动研磨刻度 probe(信息补全型)

**触发**:冷启动缺"起始研磨"时(现已改成必填按钮,理论上不再触发;保留作兜底)。
**agent 问**:你当前磨的咖啡粉有多粗?

| 选项 | 对应 |
|---|---|
| 白砂糖粗细 | 粗 |
| 食盐粗细 | 中 |
| 玉米粉粗细 | 细 |

> 补位:不给「说不清」——让用户对照实物选一个(初学者看一眼粉能判断)。这个已在冷启动表单解决,probe 兜底用同一套。

---

## P2 · E1b 酸的收尾 probe(★ 判别命门,最危险)

**触发**:用户报"酸"但没说收尾(信息不足,E1b)。
**agent 问**:这个酸,收尾是什么感觉?

| 选项 | 对应判别 |
|---|---|
| 酸完发空、收尾就没了 | → 欠萃信号(导向磨细) |
| 会化成甜、有回甘 | → 果酸口味(导向别磨、转口味) |
| 说不清 | → 继续 probe / 降置信 |

> ⚠️ **绝不能有的选项(E1b 防后门)**:任何**强度**选项——「很尖锐 / 一般酸 / 微酸 / 特别冲」。evals.md 明写:探针只能问**收尾**,不准问尖不尖锐,因为强度是噪声,高酸度豆入口也冲,会把这杯误判到欠萃边。**给了强度选项 = 在 UI 上重新打开 E1b 拼命堵的后门。**
> 这两个选项恰好对应 E1a(化甜=果酸)vs E1c(发空=欠萃)的镜像,判别力全在这一问。

---

## P3 · E12a 绝对刻度化甜 probe(萃取毕业 gate)

**触发**:四项看似全绿、用户说"有一点甜",但缺绝对毕业证据(E12a)。
**agent 问**:这点甜是什么程度?

| 选项 | 对应判别 |
|---|---|
| 甜能清楚盖过酸、喝完持续回甘 | → 绝对毕业(可进口味层) |
| 只是比最开始好一点、努力才尝到一点甜 | → 相对最优、未毕业(留萃取层) |
| 说不清 | → 继续 probe / 降置信 |

> ⚠️ **绝不能有的选项**:任何让用户**直接判"毕业了/到位了"**的选项(如「是的,酸甜平衡了」)——那等于让用户拍板 graduation,正是 E12 拼命拦的。选项只让用户**描述甜的程度**,毕业与否由 agent 从描述判,不由用户宣布。
> 对应 E12a probe 的两个语义分支(清楚盖过 vs 努力才尝到)。

---

## P4 · E11a/E11b 偏好定位 probe(口味层双分支命门)

**触发**:萃取已毕业,用户仍说"太酸不喜欢",但偏好未定位(E11 开局)。
**agent 问**:关于这个酸,更接近哪种?

| 选项 | 对应判别 |
|---|---|
| 我不太爱这类酸的风格 | → flavor_mismatch(换豆能解,E11a) |
| 我喜欢这个酸,只是想要更厚/更圆 | → taste_unaddressable(本版指路,E11b) |
| 说不清 | → 继续 probe |

> ⚠️ **绝不能有的选项**:任何指向**调冲煮参数**的选项(如「想让它没那么酸」模糊表述)——会诱导 agent 去动冻结轴。选项必须干净二分"不爱风格 vs 爱但想更厚",这正是 E11a⟂E11b 的分叉点,一对一错(换豆:E11a 对/E11b 错)。
> 这是最危险的一对:选错分支 = 把用户喜欢的好豆换掉(E11b 最严重假绿)。选项措辞必须让"爱这个酸"明确表达出来。

---

## P5 · E11c/E11d 「不甜」admission gate 反向 probe

**触发**:用户报"不甜",但不确定是欠萃还是回甘被盖(E11c/E11d 入口)。
**agent 问**:这个"不甜"更像哪种?

| 选项 | 对应判别 |
|---|---|
| 寡淡、没什么余味,酸把甜压没了 | → 欠萃,退回萃取层(E11c) |
| 有回甘的,只是酸太亮把甜盖住了、想更厚 | → 符合毕业,进口味层(E11d) |
| 说不清 | → 继续 probe |

> ⚠️ **绝不能有的选项**:直接的「不甜」笼统选项——这一问的**全部价值**就是把"不甜"消歧成"寡淡欠萃"vs"回甘被盖"。给个笼统"就是不甜"选项 = 消歧失败。
> 对应 E11c(寡淡→退萃取层)vs E11d(回甘被盖→口味层)的分叉。

---

## P6 · E7c 排气 validation probe(复现验证型)

**触发**:未知烘焙日期 + 首杯排气/流速不稳(E7c Turn 2)。
**性质特殊**:这不是"选一个描述"的 probe,是**让用户下一杯低扰动复现、再观察**的 validation probe。
**agent 说**:下一杯研磨/水温/粉量照旧,再冲一次,看流速稳定后酸空还在不在。

> ⚠️ **这个 probe 可能不需要按钮**——它要求的是"用户去再冲一杯",不是"从选项里选"。用户的回应是**下一杯的 feedback**(走 feedback 屏),不是即时选项。
> **待你判断**:P6 是否需要 options?我倾向**不需要**——它的"回应"是下一杯冲煮反馈,走正常 feedback 流,不是 probe 按钮。如果这样,P6 不进 options 库,agent 正常发文字说明即可。

| probe | 需要按钮? | 选项数 | 判别命门 |
|---|---|---|---|
| P1 冷启动研磨 | 是(兜底) | 3 | 信息补全 |
| P2 酸收尾 | 是 | 2+说不清 | ⚠️ 禁强度选项 |
| P3 绝对刻度甜 | 是 | 2+说不清 | ⚠️ 禁"直接判毕业"选项 |
| P4 偏好定位 | 是 | 2+说不清 | ⚠️ 禁调参选项,爱酸要明确 |
| P5 不甜消歧 | 是 | 2+说不清 | ⚠️ 禁笼统"不甜"选项 |
| P6 排气复现 | **否**(走 feedback) | — | 回应是下一杯反馈 |

## 实现方案(已定 · 甲-2′ 纯前端)

**probe 类型判别 = flag + 原始感官数组:**

| probe flag(record_cup) | 补充信号 | → probe 类型 |
|---|---|---|
| `preference_unspecified` | — | P4 偏好 |
| `absolute_extraction_uncertain` | — | P3 绝对刻度 |
| `degas_signals_observed` | — | P6 排气(走 feedback,不给按钮) |
| `info_insufficient` | 原始感官数组含「酸」不含「不甜/薄」 | P2 酸收尾 |
| `info_insufficient` | 原始感官数组含「不甜/薄」类 | P5 不甜消歧 |

**三个前端改动(纯前端,不动后端):**

1. **存住原始感官数组**:feedback 提交时,把用户**原始选的结构化感官词数组**(如 `["酸"]`)存进前端状态(全局 state 或 localStorage 当前轮),**别只留 record_cup 里 agent 归一化的字符串**。probe 判别读这个原始数组,不读 record_cup 的模糊字符串。

2. **flag + 原始数组判 probe 类型 → 渲染对应 options 按钮**:按上表映射。前三类 flag 直接判;`info_insufficient` 用原始感官数组拆 P2/P5。

3. **歧义兜底**:若原始数组同时含「酸」和「不甜/薄」(probe 对象歧义),**合并显示 P2+P5 两组选项**让用户选。不卡死,宁可多两个按钮。

> **为什么不从 record_cup 的 sensory 字符串判**:那是 agent 归一化后的自由字符串(如 `"不甜;酸挺明显"`),措辞不固定、可能酸和不甜混在一起,靠字符串匹配脆弱。原始 checkbox 数组是干净结构化信号,可靠。

> **已知局限(接受)**:用户同时报酸+不甜时,前端无法确知 agent 实际 probe 的是哪个对象(那是 agent 决定的,前端读不到)。此时用合并选项兜底。若试用反馈显示歧义频繁、体验差,再考虑「甲-1:agent 输出 probe_type」(动后端+重跑 eval)。当前选最小代价方案先通。

## probe 回应回传(闭合循环)

用户点了 probe 选项按钮后,前端要把答案**组装成自然语言**发回 agent(和 nlAssembly 同理),让循环往下走。例:
- P2 用户点「酸完发空、收尾就没了」→ 前端发「这酸酸完发空、收尾就没了」给 agent
- agent 收到 → 判欠萃 → 继续
> 复用 nlAssembly 的自然语言组装原则(不发结构化 payload,发整句),保持和 eval 输入同构。
