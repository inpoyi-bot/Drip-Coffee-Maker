# Future scope · Ending 路由层与其余出口

> **不属于当前 capstone 提交或 feature freeze 范围。** 当前交付只承诺 V60、研磨单轴的 happy-path 收束态。本文保留后续产品探索，不应据此宣称当前已实现以下出口。

## Task 2. Ending 路由层 + 其余出口

**归属**:在第一张卡(satisfied 收束态)之上,增加一个中心路由,按末杯 `terminate_reason` 分发到八种结局,并实现除 satisfied 外的其余出口。**所有结局共享同一容器(轨迹屏收束态),无独立屏。**

### Intent

第一张卡只处理 `satisfied` 且自带触发判断。现在有八种结局(七个 reason + 兜底),全部在轨迹屏末杯这一个判断点分发。需要一个中心分发器:读末杯 `terminate_reason`,渲染对应出口。所有出口只读轨迹、不写记录、不编造、不泄漏内部隐喻。

### Input contract

沿用第一张卡已核定的真实 schema:`cup_no` / `turn_type` / `gradient` / `direction` / `terminate_reason` / `grind_now` / `step`;轨迹为 `StoredCup[]`(`localStorage["coffee-coach:cups"]`)。本卡不新增后端字段。

**单一分流点**:轨迹屏渲染时,读末杯(`turn_type=terminate` 那杯)的 `terminate_reason`,按分发表路由。非终止态(末杯非 terminate)不渲染任何收束态,轨迹屏照常显示进行中的逐杯记录。

### 分发表

|末杯 `terminate_reason`|出口类型|能力叙事|坐标卡|
|---|---|---|---|
|`satisfied`|happy(第一张卡已实现,**不改**)|有|有(末杯 grind_now)|
|`would_overextract`|happy·认峰回退|有(末段原子不同)|有(**峰杯** grind_now)|
|`flavor_mismatch`|类别1·成功转选豆|有|无|
|`taste_unaddressable`|类别1·成功·本版够不着|有|无|
|`axis_limit_underextracted`|类别2·没完成·研磨到头|有(结尾转折)|无|
|`plateau_ambiguous`|第三态·诚实说不清|无|无|
|`axis_unreliable`|第四态·换磨劝退|无|无|
|其他/未知/null/坐实值|兜底·藏原因留过程|无|无|

> **坐实值**(`plateau_axis_topped` / `plateau_bean_decay` 等历史遗留可存储枚举)一律走**兜底**,**绝不翻译或展示为“原因”**——它们是被 E5 禁止的坐实归因。

### 通用渲染规则(所有出口共享)

1. **轨迹趋势图**:仅当可比杯 ≥ 2 时渲染;< 2 杯(如 axis_unreliable 通常只有一杯)不画趋势图(单点无意义),但逐杯记录照常。
2. **逐杯记录**:所有出口一律保留,默认折叠、一键完整展开、append-only 零删减。
3. **能力叙事**(仅“有”的出口):复用第一张卡的 `deriveSatisfiedEnding` / `phraseForCup` / `foldPhrases` 派生逻辑;映射表和折叠规则不变(连续相同折叠、拐点永不折叠)。
4. **坐标卡缺失降级**:凡“有坐标”的出口,若 `grind_now` 为空 → 隐藏坐标卡,能力叙事和收口照常(第一张卡已定的降级行为,沿用)。

### 各出口具体内容

**`would_overextract`(相对 satisfied 仅两处不同,其余全同):**

1. 能力叙事末段原子:从“到位收手”改为 **“发现再调会变差、退回上一杯最好的点、收手”**。
2. 坐标卡 `grind_now` 读**回退到的峰杯**(被认定为最优的、末杯之前那杯),**不是末杯(过头那杯)**。若 record 里无法可靠定位峰杯,见 Halt。

**`flavor_mismatch`(逐字):**

> [块1能力叙事,保留]  
> **状态:萃取到位了,剩下的是口味的事。**  
> 你这包豆的萃取已经调到位了——酸干净、有回甘,四项都到了。这一步你做成了。  
> 剩下的“还是不太喜欢”,不是萃取的问题,而是这支豆的风味取向不太对你的口味——这个冲煮端调不动。想解决,方向是换一支风味不同的豆。口味没有标准答案,你现在知道该往哪儿找了。

**`taste_unaddressable`(逐字):**

> [块1能力叙事,保留]  
> **状态:萃取到位了,这支豆很对你。**  
> 你这包豆的萃取已经调到位了,而且你喜欢它的酸——这支豆是对的,别换。  
> 你想要的“更厚一点”,不在研磨这件事上,而在水温、接触时间、浓度那些地方——这一版我只调研磨,专注把这一件事帮你走到位。想往更厚走,是下一步可以探索的方向。你的方向没错,这支豆也值得留着。

**`axis_limit_underextracted`(逐字,结尾转折):**

> [块1能力叙事,保留]  
> **状态:调整研磨能做的,做到头了。**  
> 这包豆你一路调研磨、一路在变好——从最开始的尖酸发空,到现在酸干净了很多、也能尝到一点甜。这套“看反馈、往好的方向调”的判断,你练到了。  
> 但到这一步,再磨细已经改变有限了:磨到更细会开始堵、细粉变多、流速不稳,而这杯离“酸能干净化成甜、喝完明显回甘”还差一点。光靠调研磨,这包豆到这儿就是上限了。  
> 这不是你的手法问题,是只调研磨这一件事能给的,就到这里。想再往前,得靠研磨以外的调整——那是这一版没做的部分,但你现在知道卡点在哪了。

**`plateau_ambiguous`(逐字,无能力叙事):**

> **状态:到这儿了,我们停。**  
> 这包豆连着两杯没再变好,而且开始有点纸板感、香气也淡了。  
> 这种情况有两种可能:一种是研磨这件事已经做到头了;另一种是这包豆过了最佳赏味期、状态在走下坡。说实话,单看现在的信息,我没法确定是哪一种。  
> 但这两种情况有个共同点:继续磨细都帮不了你——如果是前者,再细只会过萃发苦;如果是豆子状态的问题,磨细也补不回已经淡掉的香气。  
> 所以我们在这儿停,不是因为调好了,而是因为再调下去对这包豆已经没有意义了。这不是你的手法问题,是现实条件决定的。

**`axis_unreliable`(逐字,无能力叙事):**

> **状态:先换磨豆机,我们再一起把这包豆调到位。**  
> 你用的是砍豆机(靠刀片打碎的那种)。它磨出来的粉,粗细会两极分化——细的那部分萃过头发苦,粗的那部分没萃够发酸,同一杯里又酸又苦是这么来的,不是你冲得不对。  
> 我帮你调咖啡的办法,是每次只动一点点,再看这一杯比上一杯变好还是变坏,一步步逼近你的最佳点。但砍豆机每次磨出来的粗细都不太稳定,调了也分不清是调的作用还是磨的随机——没有稳定的反馈,这套方法就跑不起来。  
> 所以在换磨之前,我先不让你瞎调浪费豆子。换一台锥刀或平刀的磨豆机(手摇的也可以),磨出来的粉稳定了,我就能一步步帮你把这包豆调到位。这不是你的问题,是工具还没到位。

**兜底(逐字,无能力叙事,无原因):**

> **状态:这包豆的调整到这儿结束了。**  
> 下面是你这几杯完整的调整过程和轨迹,可以回看每一步。

### Must-not(交卡前重点审)

1. **不改第一张卡的 satisfied 实现**——只在其外包路由 + 增加其余出口。
2. **不写任何记录**——全部出口只读轨迹派生,禁止落账、禁止新增聚合记录。
3. **兜底绝不编原因**——未知/坐实/null,禁止翻译、展示、推断任何“为什么结束”。只陈述“结束了”+ 留过程。坐实值(`plateau_axis_topped` 等)当未知处理,**绝不渲染成“轴到顶了”之类归因**。
4. **坐标卡只出现在 `satisfied` 和 `would_overextract`**——其余出口一律无坐标(它们要么问题不在萃取、要么萃取没完成,给“落点坐标”是错的)。
5. **`would_overextract` 坐标读峰杯,不是末杯**。
6. **`plateau_ambiguous` 绝不坐实单一归因**——两种可能必须并列不定,禁止写成“你的豆不新鲜了”。
7. **`taste_unaddressable` 绝不建议换豆/转 Bean Scout,绝不擅自提粉量/调比例/水温**。
8. **`flavor_mismatch` 与 `taste_unaddressable` 绝不串味**——前者换豆、后者别换。
9. **文案逐字锁定**,无 emoji,无“轴/hill-climb/收敛/双峰分布”等内部词。
10. **无 CTA 按钮**——所有出口都不加引导按钮。
11. **不调用 LLM**——所有分发和文案确定性。
12. **趋势图 <2 杯不渲染**,但逐杯记录照常。

### Halt protocol

- **峰杯定位**:若 `would_overextract` 的“回退到的峰杯”在 record 里无法可靠定位(无字段标记哪杯是峰,只能推断)→ **停,报出 record 里实际能拿到什么,等确认**,不要猜。
- **映射表覆盖**:若某“有能力叙事”出口的轨迹里出现映射表覆盖不到的 per-cup 状态 → **停,报出该状态,等确认**,不要自造短语。
- **未知 reason 的真实分布**:实现中若发现除分发表七个值之外,实际数据里还频繁出现某个特定 reason(不只是理论遗留值)→ **停,报出它,等确认**是否需要单独出口,不要默默吞进兜底。
