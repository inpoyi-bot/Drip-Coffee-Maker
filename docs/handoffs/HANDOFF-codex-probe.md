# Handoff · Codex:probe 回应通道(纯前端,不动后端)

> 配套 `probe-options-draft.md`(选项清单 + 判别红线)必须一并读。
> **核心约束:纯前端改动。不动后端、不改 agent、不碰 eval。** options 走展示层,record_cup 结构不变。
> **背景**:当前 agent 发 probe 时,前端只显示问话,没有回应通道 → 核心收敛循环断头。任何试用者触发 probe 就卡死。这是 P0。

---

## 任务:让用户能回应 agent 的 probe(渲染成按钮)

### 现状
- agent 发 probe(turn_type=probe)时,前端显示了追问文本,但**没有回应输入**。
- feedback 屏是填冲煮反馈的,语义不对,不能用来答 probe。

### 目标
probe 出现时,前端**根据 probe 类型渲染对应的选项按钮**,用户点选后**组装成自然语言发回 agent**,循环继续。

---

## 三步实现

### 步骤1 · 存住原始感官数组(前置,必须先做)
**问题**:用户原始选的 sensory 数组(如 `["酸"]`)现在只在 feedback.tsx 的 React state,提交后被组装成自然语言就丢了;record_cup 里只有 agent 归一化的模糊字符串(如 `"不甜;酸挺明显"`)。
**改**:feedback 提交时,把**用户原始选的结构化感官词数组**存进前端状态(全局 state 或 localStorage 当前轮记录),供 probe 判别读取。**不要**依赖 record_cup 的归一化字符串做判别。

### 步骤2 · 判 probe 类型 → 渲染对应 options
probe 出现时,按「flag + 原始感官数组」判类型,渲染对应按钮(选项文案见 probe-options-draft.md,**逐字照抄,尤其红线不能违反**):

| record_cup flag | 补充信号 | probe 类型 | 按钮 |
|---|---|---|---|
| `preference_unspecified` | — | P4 偏好 | 不爱这类酸的风格 / 爱这个酸但想更厚 / 说不清 |
| `absolute_extraction_uncertain` | — | P3 绝对刻度 | 甜清楚盖过酸持续回甘 / 只是好一点努力才尝到 / 说不清 |
| `info_insufficient` | 原始数组含「酸」不含不甜/薄 | P2 酸收尾 | 酸完发空收尾没了 / 会化成甜有回甘 / 说不清 |
| `info_insufficient` | 原始数组含「不甜/薄」类 | P5 不甜消歧 | 寡淡没余味酸压甜 / 有回甘被酸盖住想更厚 / 说不清 |
| `degas_signals_observed` | — | P6 排气 | **不给按钮**,走 feedback 流(提示用户下一杯照旧再冲) |

**歧义兜底**:若原始数组**同时含「酸」和「不甜/薄」**,合并显示 P2+P5 两组选项让用户选。不卡死。

⚠️ **红线(违反即错,见 probe-options-draft.md)**:
- P2 **绝不**给强度选项(尖锐/很酸/微酸)——违反 E1b 防后门
- P3 **绝不**给"直接判毕业"选项(酸甜平衡了)——违反 E12 graduation gate
- P4 **绝不**给调参选项;"爱这个酸"必须明确表达
- P5 **绝不**给笼统"不甜"选项——消歧失败

### 步骤3 · 回应回传(闭合循环)
用户点选后,把答案**组装成自然语言整句**发回 agent(复用 nlAssembly 原则,不发结构化 payload):
- 例:P2 点「酸完发空、收尾就没了」→ 发「这酸酸完发空、收尾就没了」
- agent 收到 → 继续判断 → 循环往下

---

## 验收(端到端,后端现已可跑)

- [ ] 触发 P2(报「酸」不说收尾)→ 出现酸收尾两选项按钮,**无强度选项**
- [ ] 触发 P4(萃取毕业后报太酸)→ 出现偏好两选项,"爱这个酸想更厚"明确
- [ ] 点选后答案以自然语言发回,agent 继续,不卡死
- [ ] 歧义(酸+不甜都选)→ 合并选项兜底,不崩
- [ ] P6 排气 probe → 不出按钮,提示走下一杯反馈
- [ ] 全程不动后端、record_cup 结构不变
- [ ] typecheck / build 通过
