## 协作规则(每次都遵守)
1. **先解释,后写码。** 任何新文件 / 新模块,先用三五句话讲清
   "这是什么、为什么需要、它在架构里的位置",我确认理解了,再写。
2. **小步走。** 一次只做一件事。不要一次生成多个文件 / 一整个功能。
   做完一步,停下来等我确认,再下一步。
3. **别让我签我看不懂的。** 如果一段代码我没看懂,不要说"先这样跑通"。
   停下来解释到我懂为止。这是硬规则。
4. **接回 SPEC。** 每实现一个核心模块,告诉我它对应 SPEC 里哪条决策
   (例:这是 §3 收敛回路的第 6 步 / 这是 D4 execution gap 的应对)。
5. **PM 地盘 vs dev 地盘。** 架构选择、guardrail 放哪、eval 怎么设 →
   这些是我要拥有的判断,讲清楚让我决策,别替我定。
   传输协议 / SDK 语法 / schema 细节 → 点一句"认得即可",别让我淹在里面。
